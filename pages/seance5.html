01

SEO — introduction

Le SEO (Search Engine Optimization — référencement naturel) est l'ensemble des techniques qui permettent à votre page d'apparaître en bonne position dans les résultats de recherche. Le HTML est la fondation du SEO : un balisage correct et sémantique est la première chose qu'un moteur comme Google analyse.

📚
Analogie — Le bibliothécaire et le catalogueur

Google est un bibliothécaire automatique. Votre HTML est la fiche de catalogage de votre livre. Plus la fiche est précise (titre correct, résumé clair, mots-clés pertinents), plus le bibliothécaire rangera votre livre dans les bonnes sections et le recommandera aux bons lecteurs. Une fiche mal remplie = livre introuvable.

🕷️
Exploration
Un robot (crawler) parcourt le web, suit les liens et découvre vos pages
📊
Indexation
Google analyse votre HTML, extrait le contenu et le stocke dans son index
🏆
Classement
L'algorithme calcule votre position pour chaque requête selon 200+ critères

Ce que Google voit dans votre HTML

Google n'exécute pas toujours le CSS ou le JavaScript. Ce qu'il lit en priorité : le contenu texte brut, les balises <title>, <meta description>, les titres <h1>–<h6>, les textes alternatifs alt, les URL des liens et la structure sémantique des balises HTML5.

02

Balises meta SEO — la tête de votre document

La balise <title> et les balises <meta> dans le <head> sont le formulaire d'inscription de votre page dans l'index de Google. Ce sont les premières informations lues par les robots.

HTML
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

  <!-- ── SEO fondamental ──────────────────────────────── -->

  <!-- Titre affiché dans l'onglet ET dans Google (50–60 car.) -->
  <title>Apprendre le HTML — Cours complet pour débutants | MonSite</title>

  <!-- Description visible sous le titre dans Google (120–158 car.) -->
  <meta name="description"
        content="Cours HTML complet en 6 séances progressives : structure, listes,
                 tableaux, formulaires, multimédia et accessibilité. Exemples de
                 code copiables, exercices pratiques et glossaire.">

  <!-- Mots-clés (impact SEO faible aujourd'hui, mais bonne doc interne) -->
  <meta name="keywords" content="HTML, CSS, tutoriel, web, débutant, cours">

  <!-- Auteur de la page -->
  <meta name="author" content="Prénom Nom">

  <!-- URL canonique — évite le contenu dupliqué -->
  <link rel="canonical" href="https://monsite.com/cours-html/">

  <!-- ── Instructions pour les robots ──────────────────── -->

  <!-- Indexer la page ET suivre ses liens (comportement par défaut) -->
  <meta name="robots" content="index, follow">

  <!-- Ne PAS indexer (ex : page admin, brouillon, page de confirmation) -->
  <!-- <meta name="robots" content="noindex, nofollow"> -->

  <!-- ── Mobile et PWA ──────────────────────────────────── -->
  <meta name="theme-color" content="#0D1117">
  <link rel="icon" href="/favicon.ico" sizes="any">
  <link rel="icon" href="/icon.svg" type="image/svg+xml">
  <link rel="apple-touch-icon" href="/apple-touch-icon.png">
</head>

Règles d'or pour le <title> et la meta description

ÉlémentLongueur idéaleÀ faireÀ éviter
<title> 50–60 caractères Mot-clé principal en tête, marque à la fin, unique par page Duplication entre pages, titre trop court ou trop générique ("Accueil")
meta description 120–158 caractères Résumé vendeur avec appel à l'action, mot-clé naturellement intégré Description identique sur plusieurs pages, keyword stuffing
📌
Google peut ignorer votre meta description

Google se réserve le droit de réécrire la description affichée dans les résultats s'il estime qu'un extrait de votre page est plus pertinent pour la requête. Mais fournir une bonne description reste important : elle est utilisée quand le contenu de la page est jugé suffisamment pertinent.

03

Open Graph & Twitter Cards — partager sur les réseaux

Quand un lien est partagé sur Facebook, LinkedIn, WhatsApp ou Twitter/X, ces plateformes lisent des balises <meta> spéciales pour générer une prévisualisation riche (image, titre, description). Sans elles, vous obtenez un aperçu vide ou générique.

Open Graph (OG) — Facebook, LinkedIn, WhatsApp…

Le protocole Open Graph a été créé par Facebook en 2010. Il est aujourd'hui le standard universel pour les aperçus de liens sur presque toutes les plateformes sociales.

HTML
<head>
  <!-- ── Open Graph — balises obligatoires ──────────── -->
  <meta property="og:title"       content="Apprendre le HTML — Cours complet">
  <meta property="og:description" content="6 séances progressives pour maîtriser le HTML, de la structure aux formulaires.">
  <meta property="og:image"       content="https://monsite.com/images/og-html-cours.jpg">
  <meta property="og:url"         content="https://monsite.com/cours-html/">
  <meta property="og:type"        content="website">

  <!-- ── Open Graph — balises recommandées ─────────── -->
  <meta property="og:locale"      content="fr_FR">
  <meta property="og:site_name"   content="Cours HTML MonSite">
  <meta property="og:image:width"  content="1200">
  <meta property="og:image:height" content="630">
  <meta property="og:image:alt"    content="Aperçu du cours HTML — fond sombre, code en surbrillance">

  <!-- ── Twitter / X Cards ─────────────────────────── -->
  <meta name="twitter:card"        content="summary_large_image">
  <meta name="twitter:site"        content="@moncompte_twitter">
  <meta name="twitter:creator"     content="@auteur_twitter">
  <meta name="twitter:title"       content="Apprendre le HTML — Cours complet">
  <meta name="twitter:description" content="6 séances progressives pour maîtriser le HTML.">
  <meta name="twitter:image"       content="https://monsite.com/images/twitter-html-cours.jpg">
  <meta name="twitter:image:alt"   content="Aperçu du cours HTML">
</head>

Structure sémantique pour le SEO

La sémantique HTML (vue en séance 2) n'est pas seulement une question de propreté de code : elle a un impact direct sur le SEO. Google comprend mieux une page dont la structure est explicite, et peut en extraire des informations enrichies pour les résultats.

Les signaux sémantiques les plus importants pour Google

Signal HTMLImpact SEORègle à suivre
<h1>⭐⭐⭐⭐⭐ Très fortUn seul <h1> par page, contenant le mot-clé principal naturellement
<h2>–<h6>⭐⭐⭐⭐ FortHiérarchie logique, mots-clés secondaires, pas de sauts (h1 → h3)
alt des images⭐⭐⭐⭐ FortDescriptif, naturel, contient le sujet de l'image — jamais vide si image informative
Texte des liens <a>⭐⭐⭐⭐ FortDescriptif ("lire le guide HTML") jamais générique ("cliquez ici")
<main>⭐⭐⭐ MoyenDélimite le contenu principal — aidez Google à le distinguer de la nav/footer
<article>⭐⭐⭐ MoyenContenu autonome (article de blog, news) — peut être partagé indépendamment
<strong>, <em>⭐⭐ FaibleSignale l'importance — à utiliser avec parcimonie et pour du sens réel
URL lisible + canonical⭐⭐⭐⭐ FortURLs courtes, en français, sans paramètres inutiles

Structured data — données structurées JSON-LD

Les données structurées permettent à Google d'afficher des rich snippets dans les résultats : étoiles de note, prix de produits, horaires d'événements, recettes… Le format recommandé par Google est JSON-LD, placé dans une balise <script> dans le <head>.

HTML
<head>
  <!-- ── Données structurées JSON-LD (schema.org) ────── -->
  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "Course",
    "name": "Apprendre le HTML — Cours complet",
    "description": "Cours HTML en 6 séances progressives.",
    "provider": {
      "@type": "Organization",
      "name": "MonSite",
      "url": "https://monsite.com"
    },
    "url": "https://monsite.com/cours-html/",
    "inLanguage": "fr",
    "educationalLevel": "Débutant"
  }
  </script>

  <!-- ── Données structurées pour un article de blog ─── -->
  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "Article",
    "headline": "Les 10 meilleures pratiques HTML5",
    "author": { "@type": "Person", "name": "Alice Dupont" },
    "datePublished": "2024-09-15",
    "dateModified": "2024-10-01",
    "image": "https://monsite.com/images/article-html5.jpg"
  }
  </script>
</head>
05

Accessibilité — pourquoi et pour qui ?

L'accessibilité web (notée a11y — "a" + 11 lettres + "y") est la pratique qui consiste à rendre les sites web utilisables par toutes les personnes, quelles que soient leurs capacités physiques, cognitives ou techniques.

♿
Analogie — Le trottoir avec abaissement de bordure

Un trottoir abaissé est conçu pour les fauteuils roulants. Mais il profite aussi aux parents avec poussette, aux livreurs avec chariot, aux cyclistes, aux enfants en vélo… L'accessibilité web fonctionne pareil : une interface accessible bénéficie à tout le monde, pas seulement aux personnes en situation de handicap.

Qui bénéficie de l'accessibilité ?

👁️ Visuel

Cécité, malvoyance, daltonisme, sensibilité à la lumière. Utilisent des lecteurs d'écran (VoiceOver, NVDA), du zoom, des contrastes élevés.

🦻 Auditif

Surdité, malentendance. Ont besoin de sous-titres, transcriptions, alternatives textuelles aux contenus audio.

🖐️ Moteur

Paralysie, tremblement, SEP. Naviguent uniquement au clavier (Tab, Entrée, flèches) ou avec des dispositifs alternatifs.

🧠 Cognitif

Dyslexie, autisme, TDAH, troubles de la mémoire. Ont besoin de structure claire, d'un langage simple, d'animations réduites.

Le référentiel WCAG

Les WCAG (Web Content Accessibility Guidelines) sont les lignes directrices internationales publiées par le W3C. Elles s'organisent autour de 4 principes POUR :

PrincipeSignificationExemples
PerceptibleLes informations doivent être présentables de plusieurs façonsTexte alt pour les images, sous-titres pour vidéos, contraste suffisant
OpérableL'interface doit être navigable sans sourisNavigation clavier complète, focus visible, pas de piège à focus
UnderstandableLe contenu doit être lisible et prévisibleLangue déclarée (lang="fr"), labels de formulaires, messages d'erreur clairs
RobustCompatible avec les technologies d'assistance actuelles et futuresHTML valide, ARIA correct, pas de dépendance à une seule technologie
⚖️
Accessibilité et obligation légale

Dans l'Union Européenne, la directive EAAA (European Accessibility Act) impose des normes d'accessibilité à tous les services numériques commerciaux depuis juin 2025. En France, la RGAA (Référentiel Général d'Amélioration de l'Accessibilité) est obligatoire pour les services publics. En pratique : même si la loi ne vous oblige pas encore, vos utilisateurs l'exigent.

ARIA — Accessible Rich Internet Applications

ARIA est un ensemble d'attributs HTML qui permettent de compléter ou corriger la sémantique d'une page pour les technologies d'assistance (lecteurs d'écran). La règle d'or : n'utilisez ARIA que si le HTML natif ne suffit pas.

⚠️
La première règle d'ARIA — préférez le HTML natif

Avant d'ajouter un role="button" sur un <div>, demandez-vous : pourquoi ne pas simplement utiliser <button> ? Un vrai <button> est focusable au clavier, activable à Entrée et Espace, et annoncé correctement — gratuitement. ARIA ne recrée pas le comportement, il ne fait qu'annoter.

Les 3 types d'attributs ARIA

HTML
<!-- ── 1. RÔLES — définir le type d'élément ──────────── -->
<div role="navigation" aria-label="Menu principal">…</div>
<div role="main">…</div>
<div role="dialog" aria-modal="true" aria-labelledby="titre-modal">…</div>
<span role="button" tabindex="0">Ouvrir</span> <!-- derniers recours seulement -->

<!-- ── 2. PROPRIÉTÉS — informations statiques ─────────── -->
<input type="text"
       aria-label="Rechercher dans le site"
       aria-required="true"
       aria-describedby="conseil-recherche">
<p id="conseil-recherche">Entrez un mot-clé, au moins 3 caractères.</p>

<img src="schema.png" alt="" role="presentation"> <!-- image décorative -->
<img src="graphique.png"
     alt="Graphique barres : ventes 2023"
     aria-describedby="desc-graphique">
<p id="desc-graphique">Les ventes ont augmenté de 30% entre janvier et juin.</p>

<!-- ── 3. ÉTATS — informations dynamiques ─────────────── -->
<button aria-expanded="false" aria-controls="menu-contenu">
  Menu
</button>
<nav id="menu-contenu" aria-hidden="true">…</nav>

<input type="text"
       aria-invalid="true"
       aria-errormessage="erreur-email">
<p id="erreur-email" role="alert">
  Adresse e-mail invalide.
</p>

<div role="progressbar"
     aria-valuenow="65"
     aria-valuemin="0"
     aria-valuemax="100"
     aria-label="Chargement : 65%">
</div>

Les rôles ARIA les plus utilisés

role="alert"
Message important lu immédiatement par le lecteur d'écran. Pour erreurs, confirmations.
role="dialog"
Modale/popup. Doit avoir aria-modal="true" et aria-labelledby.
role="tab"
Onglet dans un système tablist/tabpanel.
role="tooltip"
Info-bulle associée à un élément via aria-describedby.
role="status"
Message d'état non-urgent (chargement, sauvegarde auto).
role="presentation"
Élément purement décoratif — retiré de l'arbre d'accessibilité.

aria-label vs aria-labelledby vs aria-describedby

AttributSource du texteCas d'usage typique
aria-labelChaîne directe dans l'attributBouton icône seul : <button aria-label="Fermer">✕</button>
aria-labelledbyID d'un élément existant dans la pageModal référençant son titre : aria-labelledby="id-du-h2"
aria-describedbyID d'un élément de descriptionInput avec aide contextuelle visible : aria-describedby="conseil-id"

Bonnes pratiques a11y en HTML

La plupart des problèmes d'accessibilité viennent de mauvaises habitudes HTML corrigibles facilement. Voici les plus importantes, avec leurs solutions.

Navigation au clavier — indispensable

Toute l'interface doit être navigable uniquement avec le clavier : Tab avance, Shift+Tab recule, Entrée/Espace active, flèches naviguent dans les menus.

HTML
<!-- ✅ BIEN — bouton natif, focusable et activable nativement -->
<button type="button" onclick="ouvrirMenu()">Menu</button>

<!-- ❌ MAL — div non focusable, inaccessible au clavier -->
<div onclick="ouvrirMenu()">Menu</div>

<!-- Si vous DEVEZ utiliser un div (raison CSS), ajoutez tabindex et rôle -->
<div role="button" tabindex="0"
     onclick="ouvrirMenu()"
     onkeydown="if(event.key==='Enter'||event.key===' ') ouvrirMenu()">
  Menu
</div>

<!-- ── Focus visible — WCAG 2.4.7 ─────────────────────── -->
<!-- Ne JAMAIS faire : outline: none; sans alternative visible -->
<style>
  /* Contour de focus personnalisé mais toujours visible */
  :focus-visible {
    outline: 3px solid #2DD4BF;
    outline-offset: 3px;
    border-radius: 4px;
  }
</style>

<!-- ── Lien d'évitement — sauter la navigation ──────── -->
<a href="#contenu-principal" class="skip-link">
  Aller au contenu principal
</a>
<!-- Ce lien doit être le PREMIER élément du body. Il devient visible au focus. -->

Contraste de couleurs — WCAG AA

Le ratio de contraste entre le texte et son fond doit être d'au moins 4.5:1 pour le texte normal et 3:1 pour le texte grand (18pt+) ou gras (14pt+). Niveau AAA : 7:1.

❌ INSUFFISANT Texte difficile à lire
Ratio : 2.1:1 (échec AA)
✅ CONFORME AA Texte lisible
Ratio : 12.6:1 (passe AAA)
💡
Outils de vérification de contraste

WebAIM Contrast Checker (webaim.org/resources/contrastchecker/) : entrez vos couleurs en hex et voyez instantanément le ratio et si vous passez AA/AAA. Dans Chrome DevTools : inspectez un texte → propriété color → cliquez le carré couleur → le Contrast Ratio s'affiche avec ✓ ou ✗.

Formulaires accessibles

HTML
<!-- ✅ BIEN — label lié explicitement -->
<label for="prenom">Prénom <span aria-hidden="true">*</span></label>
<input type="text" id="prenom" name="prenom"
       required aria-required="true"
       autocomplete="given-name">

<!-- ✅ BIEN — message d'erreur lié et annoncé -->
<input type="email" id="email"
       aria-invalid="true"
       aria-describedby="erreur-email-msg">
<p id="erreur-email-msg" role="alert" style="color:red;">
  ⚠️ Veuillez entrer une adresse e-mail valide (ex : alice@exemple.com)
</p>

<!-- ✅ BIEN — placeholder n'est PAS un label -->
<label for="ville">Ville de résidence</label>
<input type="text" id="ville" placeholder="Ex : Paris">
<!-- placeholder disparaît à la saisie → ne jamais l'utiliser seul comme label -->

<!-- ✅ BIEN — groupe de boutons radio avec fieldset -->
<fieldset>
  <legend>Niveau d'expérience</legend>
  <label><input type="radio" name="niveau" value="debutant"> Débutant</label>
  <label><input type="radio" name="niveau" value="moyen"> Intermédiaire</label>
  <label><input type="radio" name="niveau" value="avance"> Avancé</label>
</fieldset>

Type d'imageAttribut altExemple

Langue et mouvement — autres bonnes pratiques

HTML
<!-- Déclarer la langue de la page (WCAG 3.1.1) -->
<html lang="fr">

<!-- Changer de langue dans le contenu -->
<p>Le terme anglais <span lang="en">accessibility</span> se traduit par accessibilité.</p>

<!-- ── Réduire les animations (WCAG 2.3.3) ──────────── -->
<style>
  /* Respecter la préférence système "Réduire les animations" */
  @media (prefers-reduced-motion: reduce) {
    *, *::before, *::after {
      animation-duration: 0.01ms !important;
      transition-duration: 0.01ms !important;
    }
  }
</style>

<!-- ── Liens — texte descriptif (WCAG 2.4.4) ────────── -->

<!-- ❌ MAL -->
<a href="guide.html">Cliquez ici</a>

<!-- ✅ BIEN -->
<a href="guide.html">Lire le guide HTML5 complet</a>

<!-- ✅ BIEN — si texte court, ajouter aria-label -->
<a href="guide.html" aria-label="Lire le guide HTML5 complet">En savoir plus</a>

Audit d'accessibilité — outils et méthodes

Un audit d'accessibilité évalue votre page selon les critères WCAG. Il existe des outils automatiques (qui détectent ~30–40% des problèmes) et des méthodes manuelles (indispensables pour le reste).

Outils automatiques

OutilTypeComment l'utiliser
LighthouseExtension Chrome intégréeF12 → Lighthouse → cocher "Accessibilité" → Analyser. Score /100 avec recommandations.
axe DevToolsExtension Chrome/FirefoxOutil le plus précis. F12 → onglet "axe" → Scan. Classe les problèmes par criticité.
WAVEExtension + site webwave.webaim.org ou extension. Visualise les problèmes directement sur la page.
W3C ValidatorSite webvalidator.w3.org. HTML invalide = problèmes d'accessibilité potentiels.

Audit manuel — les 5 tests incontournables

  • ⌨️
    Test clavier complet
    Débranchez votre souris. Naviguez sur toute la page uniquement avec Tab, Shift+Tab, Entrée, Espace et flèches. Vous devez pouvoir accéder à tous les liens, boutons, champs et interactifs. Le focus doit toujours être visible. Vous ne devez jamais être "piégé".
  • 🔊
    Test lecteur d'écran
    Activez NVDA (Windows, gratuit) ou VoiceOver (Mac : Cmd+F5 / iOS : 3 pressions bouton). Fermez les yeux. Naviguez uniquement en écoutant. Est-ce que le contenu a du sens ? Les images ont-elles un alt ? Les formulaires sont-ils annoncés correctement ?
  • 🔍
    Test zoom 200%
    Dans Chrome : Ctrl/Cmd + (ou dans les paramètres → zoom 200%). La page doit rester lisible et utilisable sans perte de contenu ni scrolling horizontal obligatoire. WCAG 1.4.4.
  • 🎨
    Test sans couleurs (niveaux de gris)
    Activez le mode niveaux de gris de votre OS ou utilisez l'extension "Colorblindly". Toutes les informations importantes doivent rester compréhensibles sans la couleur. Un lien ne doit pas être distinguable uniquement par sa couleur rouge.
  • 📐
    Test de structure avec les titres
    Installez l'extension "HeadingsMap" (Chrome/Firefox). Elle affiche l'arborescence de vos titres h1–h6. La structure doit être logique et hiérarchique. Un h1 unique, des h2 pour les sections principales, h3 pour les sous-sections — pas de saut de niveau.

Checklist rapide — les 10 points essentiels

CHECKLIST
□ lang="fr" (ou autre) sur la balise <html>
□ <title> unique et descriptif sur chaque page
□ Un seul <h1> par page, hiérarchie des titres cohérente
□ Tout <img> a un attribut alt (vide "" si décoratif)
□ Chaque <input> a un <label> lié (for/id)
□ Contrastes de couleurs ≥ 4.5:1 (texte normal) / 3:1 (grand texte)
□ Navigation clavier complète — focus visible sur tous les éléments interactifs
□ Pas de "cliquez ici" — texte des liens descriptif
□ @media (prefers-reduced-motion: reduce) implémenté si animations présentes
□ Page valide W3C (validator.w3.org) sans erreurs bloquantes
🔗
SEO et accessibilité — des objectifs communs

Ce n'est pas un hasard : beaucoup de bonnes pratiques SEO sont aussi des bonnes pratiques a11y. Un bon alt aide Google et les lecteurs d'écran. Un <h1> bien rédigé structure le contenu pour Google et pour la navigation par titres. Des textes de liens descriptifs aident l'indexation et la navigation clavier. Faites du bon HTML : vous gagnez sur les deux tableaux.

Exercices pratiques

EX 01 🔍 Audit et correction d'une page existante
  1. Prenez votre page créée en séance 1 (ou recréez-en une simple avec titre, navigation, image, formulaire et pied de page).
  2. Ouvrez Chrome DevTools → Lighthouse → cochez Accessibilité → Analyser. Notez le score et les problèmes listés.
  3. Installez l'extension axe DevTools et relancez un scan. Comparez les résultats avec Lighthouse.
  4. Corrigez au minimum : tout alt manquant, tout <label> manquant, les problèmes de contraste signalés.
  5. Vérifiez la navigation clavier : Tab sur tous les éléments interactifs. Le focus est-il visible ? Ajoutez du CSS de focus si nécessaire.
  6. Relancez le scan Lighthouse. Comparez votre nouveau score.
EX 02 🌐 Optimiser le head SEO d'une page
  1. Créez un fichier article.html sur un sujet de votre choix (cuisine, sport, technologie…).
  2. Écrivez un <title> de 50–60 caractères avec le mot-clé principal en début.
  3. Rédigez une <meta name="description"> de 120–158 caractères, incitative et avec le mot-clé.
  4. Ajoutez toutes les balises Open Graph obligatoires (og:title, og:description, og:image, og:url, og:type).
  5. Ajoutez les balises Twitter Card (twitter:card="summary_large_image" + title + description + image).
  6. Ajoutez un bloc de données structurées JSON-LD de type Article avec auteur, date et image.
  7. Vérifiez votre JSON-LD sur search.google.com/test/rich-results.
EX 03 ♿ Composant accessible complet — menu de navigation
  1. Créez un menu de navigation principal avec <nav aria-label="Navigation principale"> et une liste <ul>.
  2. Ajoutez un lien d'évitement <a href="#main-content" class="skip-link">Aller au contenu</a> en tout premier enfant du body. Stylez-le pour qu'il soit invisible jusqu'au focus (position: absolute; top: -100px; → top: 0 au :focus).
  3. Ajoutez un sous-menu déroulant sur l'un des items. Implémentez le pattern ARIA correct : aria-haspopup="true" sur le bouton parent, aria-expanded="false/true" mis à jour par JavaScript, aria-controls pointant vers le sous-menu.
  4. Assurez-vous que les flèches ↑/↓ naviguent dans le sous-menu et qu'Échap le ferme.
  5. Vérifiez avec WAVE ou axe que votre menu n'a pas d'erreurs d'accessibilité.

Mini-glossaire

SEO
Search Engine Optimization. Ensemble des pratiques pour améliorer la visibilité d'une page dans les résultats de recherche organiques.
Crawler
Robot automatique qui parcourt le web en suivant les liens. Google Bot est le crawler de Google.
Canonical
Balise <link rel="canonical"> indiquant l'URL de référence d'une page. Évite la pénalité pour contenu dupliqué.
Open Graph
Protocole de métadonnées (meta property="og:*") créé par Facebook pour générer des aperçus de liens sur les réseaux sociaux.
Rich Snippet
Résultat enrichi dans Google affichant des étoiles, prix, dates… grâce aux données structurées schema.org.
a11y
Abréviation d'accessibility : "a" + 11 lettres + "y". Désigne l'accessibilité numérique.
WCAG
Web Content Accessibility Guidelines. Référentiel international du W3C. Niveaux A, AA (cible standard), AAA.
ARIA
Accessible Rich Internet Applications. Attributs HTML (role, aria-*) pour annoter la sémantique en faveur des technologies d'assistance.
Lecteur d'écran
Logiciel qui lit le contenu d'un écran à voix haute. Ex : NVDA (Windows, gratuit), VoiceOver (Mac/iOS), TalkBack (Android).
Ratio de contraste
Rapport entre la luminosité relative du texte et de son fond. WCAG AA exige 4.5:1 (texte normal) et 3:1 (grand texte).
RGAA
Référentiel Général d'Amélioration de l'Accessibilité. Déclinaison française des WCAG, obligatoire pour les services publics.
skip link
Lien "Aller au contenu principal" caché visuellement mais premier dans le code — permet aux utilisateurs clavier d'éviter de traverser la navigation à chaque page.
