01

Meta tags pour le SEO

Le SEO (Search Engine Optimization) commence dans le <head>. Les balises <meta> donnent aux moteurs de recherche les informations essentielles sur votre page.

HTML — Meta SEO 
<head>
  <meta charset="UTF-8">
  <meta name="viewport"
    content="width=device-width, initial-scale=1.0">

  <!-- Titre : 50-60 caractères -->
  <title>Cours HTML Complet - Apprenez le HTML5
    | Mon Site</title>

  <!-- Description : 150-160 caractères -->
  <meta name="description"
    content="Apprenez le HTML5 de zéro. Cours
    complet avec exercices, exemples et
    projets pratiques pour débutants.">

  <!-- Indexation -->
  <meta name="robots"
    content="index, follow">

  <!-- Auteur -->
  <meta name="author"
    content="Jean Dupont">
</head>
RENDU
Cours HTML Complet - Apprenez le HTML5 | Mon Site
https://monsite.com/
Apprenez le HTML5 de zéro. Cours complet avec exercices, exemples et projets pratiques pour débutants.

Aperçu de votre site dans Google — le titre et la description sont ce que voient les utilisateurs.

MetaRôle SEO
titleTitre cliquable dans Google. 50-60 car. Mot-clé principal en premier.
descriptionTexte sous le titre dans Google. 150-160 car. Donne envie de cliquer.
robotsindex = page dans Google. follow = suivre les liens. noindex = exclure.
02

Open Graph (Facebook, LinkedIn)

Le protocole Open Graph (OG) contrôle l'apparence de votre page quand elle est partagée sur Facebook, LinkedIn, Pinterest, etc. Sans OG, le réseau social choisit une image au hasard.

HTML — Open Graph 
<head>
  <meta property="og:title"
    content="Cours HTML Complet">
  <meta property="og:description"
    content="Apprenez le HTML5 de zéro.">
  <meta property="og:image"
    content="https://monsite.com/
    images/og-html-cours.jpg">
  <meta property="og:url"
    content="https://monsite.com/">
  <meta property="og:type"
    content="website">
  <meta property="og:locale"
    content="fr_FR">
  <meta property="og:site_name"
    content="Mon Site">
</head>
RENDU
📷 og:image (1200×630 px)
monsite.com
Cours HTML Complet
Apprenez le HTML5 de zéro.

Aperçu du partage sur Facebook/LinkedIn.

💡
Taille recommandée pour og:image

L'image doit faire au moins 1200×630 pixels, moins de 8 Mo. Utilisez le validateur officiel : developers.facebook.com/tools/debug pour tester l'affichage.

03

Twitter Cards

Les Twitter Cards fonctionnent comme Open Graph mais pour X (Twitter). Deux types principaux : summary (titre+description+image) et summary_large_image (image en grand).

HTML — Twitter Cards 
<head>
  <meta name="twitter:card"
    content="summary_large_image">
  <meta name="twitter:site"
    content="@monsite">
  <meta name="twitter:title"
    content="Cours HTML Complet">
  <meta name="twitter:description"
    content="Apprenez le HTML5 de zéro.">
  <meta name="twitter:image"
    content="https://monsite.com/
    images/twitter-card.jpg">
</head>
RENDU
📷 twitter:image (summary_large_image)
Cours HTML Complet
Apprenez le HTML5 de zéro.
📌
OG et Twitter : les deux sont nécessaires

Facebook/LinkedIn lisent les balises og:. Twitter/X lit twitter:. Pour être compatible partout, incluez les deux ensembles. Twitter utilise aussi og:title si twitter:title manque.

04

JSON-LD & Schema.org

JSON-LD (JavaScript Object Notation for Linked Data) est le format recommandé par Google pour les données structurées. Il permet aux moteurs de recherche d'afficher des rich snippets : étoiles, prix, recettes, FAQ.

HTML — JSON-LD 
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Course",
  "name": "Cours HTML Complet",
  "description": "Apprenez le HTML5 de zéro.",
  "provider": {
    "@type": "Organization",
    "name": "Mon Site"
  },
  "author": {
    "@type": "Person",
    "name": "Jean Dupont"
  }
}
</script>
RENDU

Les données JSON-LD sont invisibles dans le navigateur mais lisibles par Google. Elles peuvent générer des résultats enrichis dans les SERP.

Testez vos données structurées avec : search.google.com/test/rich-results

💡
Types Schema.org courants

Article, Course, Product, Recipe, FAQPage, Event, LocalBusiness. Chaque type a des propriétés spécifiques. Consultez schema.org pour la liste complète.

05

ARIA : rôles & attributs

ARIA (Accessible Rich Internet Applications) améliore l'accessibilité quand le HTML sémantique ne suffit pas. Règle d'or : pas d'ARIA si le HTML natif fait le travail.

HTML — ARIA 
<!-- Rôle explicite (quand le HTML sémantique
     ne suffit pas) -->
<div role="navigation" aria-label="Menu principal">
  <ul>
    <li><a href="/">Accueil</a></li>
  </ul>
</div>

<!-- aria-label : alternative au label invisible -->
<button aria-label="Fermer">×</button>

<!-- aria-hidden : cacher un élément décoratif -->
<span aria-hidden="true">★</span>

<!-- aria-describedby : description liée -->
<input type="text" aria-describedby="aide-nom">
<p id="aide-nom">Saisissez votre nom complet.</p>

<!-- aria-expanded : état d'un menu déroulant -->
<button aria-expanded="false"
  aria-controls="menu-deroulant">
  Menu
</button>
RENDU

Les attributs ARIA ne changent pas le rendu visuel — ils aident les technologies d'assistance.

← aria-label="Fermer"

Attribut ARIAUsage
aria-labelTexte alternatif pour un élément sans label visible.
aria-labelledbyRéférence l'id d'un élément qui sert de label.
aria-describedbyRéférence l'id d'une description complémentaire.
aria-hiddenCache un élément aux lecteurs d'écran (décoratif).
aria-expandedIndique si un élément est déplié (true/false).
aria-controlsIndique quel élément est contrôlé (accordéon, menu).
roleDéfinit le rôle d'un élément (navigation, button, dialog).
⚠️
Règle : pas d'ARIA inutile

N'ajoutez pas role="navigation" sur un <nav> — c'est déjà implicite. N'ajoutez pas role="button" sur un <button>. Utilisez ARIA uniquement quand le HTML natif est insuffisant.

06

WCAG — Règles d'accessibilité

Les WCAG (Web Content Accessibility Guidelines) sont le standard international d'accessibilité web. Les principes clés pour le HTML : contraste suffisant, navigation au clavier, alternatives textuelles.

🎨 Contraste

Le texte doit avoir un contraste minimum de 4.5:1 avec son fond (3:1 pour les grands textes). Les outils comme WebAIM Contrast Checker vérifient cela.

⌨️ Clavier

Toutes les fonctionnalités doivent être accessibles au clavier (Tab, Enter, Escape). Un focus visible (outline) est obligatoire — ne faites pas outline: none sans le remplacer.

🖼️ Alt

Toute image doit avoir un alt pertinent. alt="" pour les images décoratives. Jamais de alt="image" ou alt="photo.jpg".

📝 Sémantique

Utilisez les balises pour leur sens, pas leur style. Un <h1> doit être un vrai titre, pas du texte en grand. Les lecteurs d'écran naviguent par titres.

✅ Accessible

Titres hiérarchisés · alt descriptifs · navigation au clavier · contrastes suffisants · labels sur tous les champs.

❌ Non accessible

Images sans alt · titres manquants · focus invisible · contrastes insuffisants · champs sans label.

07

Lecteurs d'écran

Les lecteurs d'écran (NVDA, JAWS, VoiceOver) transforment le contenu visuel en synthèse vocale. Comprendre comment ils fonctionnent est essentiel pour produire un HTML accessible.

💡
Testez avec NVDA (gratuit)

NVDA (NonVisual Desktop Access) est un lecteur d'écran gratuit pour Windows. Téléchargez-le et naviguez sur votre site les yeux fermés pour comprendre l'expérience utilisateur.

Raccourci NVDAAction
HNaviguer entre les titres (h1→h6)
BNaviguer entre les boutons
FNaviguer entre les formulaires
LNaviguer entre les listes
TNaviguer entre les tableaux
DNaviguer entre les landmarks (ARIA)
08

Outils d'audit : Lighthouse

Lighthouse est un outil Google intégré à Chrome qui analyse une page web et donne des scores sur 100 pour : Performance, Accessibilité, SEO, Best Practices et PWA.

💡
Lancer un audit

Dans Chrome : F12 → onglet "Lighthouse" (ou "Audits") → cliquer sur "Analyser le chargement de la page". Après quelques secondes, vous obtenez un rapport détaillé avec des conseils d'amélioration pour chaque catégorie.

🚀 Performance

Temps de chargement, images optimisées, JavaScript différé. Un bon score = site rapide = meilleur SEO.

♿ Accessibilité

Contraste, labels, ARIA, structure sémantique. 100% est l'objectif pour un site inclusif.

🔍 SEO

Meta tags, title, description, balises d'en-tête, données structurées, robots.txt, sitemap.

✅ Best Practices

HTTPS, requêtes sécurisées, doctype valide, console sans erreur, images dimensionnées.

📌
Objectif : 90+ partout

Visez un score d'au moins 90 dans chaque catégorie. Lighthouse vous donne des suggestions concrètes pour chaque point négatif. C'est le meilleur outil pour améliorer votre site étape par étape.

09

Exercices pratiques

EX 01 ✏️ Audit SEO de votre page
  1. Prenez une page que vous avez créée dans les séances précédentes.
  2. Vérifiez qu'elle a : un title optimisé (50-60 car.), une meta description (150-160 car.), un h1 unique.
  3. Ajoutez les balises Open Graph (og:title, og:description, og:image, og:url).
  4. Ajoutez les Twitter Cards (twitter:card, twitter:title, twitter:description, twitter:image).
EX 02 🔧 Audit Lighthouse
  1. Ouvrez votre page dans Chrome.
  2. F12 → onglet Lighthouse → "Analyser le chargement de la page".
  3. Notez vos scores dans chaque catégorie.
  4. Corrigez au moins 3 problèmes identifiés.
  5. Relancez l'audit et comparez les scores.
EX 03 🔍 Accessibilité
  1. Vérifiez que tous vos formulaires ont des <label> avec for.
  2. Vérifiez que toutes vos images ont un alt pertinent.
  3. Ajoutez aria-label aux boutons qui n'ont qu'une icône.
  4. Vérifiez que la navigation au clavier (Tab) fonctionne correctement.
  5. Testez avec le contraste : utilisez webaim.org/resources/contrastchecker.
